# COE (Correction of Errors) Format

ai-incident-commander generates postmortems in the Amazon-style **Correction of
Errors (COE)** format. The postmortem is generated by the
`generate_postmortem_node` via an LLM call, then parsed into a structured
`Postmortem` model. Severity determines which sections are included, and every
AI-authored section is explicitly labeled so human reviewers know what to
scrutinize before publishing.

---

## Overview

The COE format focuses on **what failed** and **why**, never **who failed**.
The postmortem generation prompt enforces blameless rules as a safety
guardrail, and the output labels each section's provenance (AI-generated vs.
session data) so reviewers can apply appropriate scrutiny.

The postmortem is produced in two stages:

1. **`_build_postmortem_prompt`** assembles the timeline, stakeholder updates,
   and top-3 reranked evidence items into a prompt with blameless rules and
   severity-conditional section instructions.
2. **`_parse_postmortem_response`** parses the LLM response into a `Postmortem`
   object, handling section headers, action items (`description | owner, priority`
   format), severity-conditional sections, and MTTR calculation.

If the LLM call fails, the graph continues with empty sections showing
"insufficient data" placeholders so the human reviewer can fill in details.

---

## Severity-Conditional Sections

The number of sections depends on incident severity. Lower-severity incidents
omit sections that aren't relevant.

| Severity | Sections | Omitted Sections                                    |
|----------|----------|-----------------------------------------------------|
| SEV1     | 8        | (none — all sections present)                       |
| SEV2     | 6        | Stakeholder Communication Log, Regulatory/Compliance Impact |
| SEV3     | 5        | Customer Impact, Stakeholder Communication Log, Regulatory/Compliance Impact |

### Always-present sections (all severities)

1. Summary
2. Timeline
3. Root Cause Analysis
4. Systemic Contributing Factors
5. Action Items

### SEV1/SEV2 only

- Customer Impact

### SEV1 only

- Stakeholder Communication Log
- Regulatory/Compliance Impact

---

## AI Section Labeling Convention

Every section in the postmortem is tagged with its provenance so human
reviewers know what to trust:

- **`[AI-GENERATED — review carefully]`** — the section content was produced by
  the LLM and has not been human-verified. Review for factual accuracy and
  blameless compliance before publishing.
- **`[From session data]`** — the section was reconstructed from structured
  state data (timeline events, update logs), not LLM hallucination.

The `PostmortemSection` model carries an `ai_generated: bool` field (default
`True`) that controls which label is applied. The following sections are always
marked as `ai_generated=False` because they are reconstructed from session
state:

- **Timeline** — rebuilt from structured `TimelineEvent` records
- **Stakeholder Communication Log** — rebuilt from sent `StakeholderUpdate` records

All other sections default to `ai_generated=True`.

A summary table appears at the bottom of `postmortem.md` listing each section
and its source:

```markdown
### AI Section Labels
| Section | Source |
|---------|--------|
| Summary | AI-generated |
| Timeline | Session data |
| Root Cause Analysis | AI-generated |
| Systemic Factors | AI-generated |
| Customer Impact | AI-generated |
| Stakeholder Comm Log | Session data |
| Regulatory Impact | AI-generated |
```

## Blameless Rules

The LLM prompt includes explicit blameless rules as a safety guardrail:

```
BLAMELESS RULES (SAFETY GUARDRAIL):
- Focus on what failed, not who failed.
- Systemic Contributing Factors must focus on processes, not people.
- Do not include individual names in root cause or systemic factors.
```

These rules are injected into every postmortem generation prompt. The
`Systemic Contributing Factors` section in particular must address processes,
tooling, and systemic gaps — never individual performance or named people.

Action items may reference team names as suggested owners (e.g. "Platform
Team", "SRE Team"), but root cause and systemic factors sections must not
include individual names.

---

## The 8 Sections

### 1. Summary

A brief incident overview. Always present. AI-generated.

Covers what happened, when it started, how long it lasted, and the blast
radius. Typically 2-4 paragraphs.

### 2. Customer Impact

Who was affected and how. Present for SEV1 and SEV2 only. AI-generated.

Quantifies the impact: number of affected users, transactions failed, revenue
at risk, geographic scope, and duration of customer-visible degradation.

### 3. Timeline

Key events in chronological order. Always present. Marked as **session data**
(not AI-generated) because it is reconstructed from structured `TimelineEvent`
records.

Includes events from all sources: alert trigger, log entries, chat messages,
GitHub PR merges, and manual notes. Deploy-correlated events are marked.

### 4. Root Cause Analysis

What caused the incident. Always present. AI-generated.

Explains the technical root cause — the specific failure mechanism that
produced the observable symptoms. Should trace from the alert back to the
underlying cause (e.g. a config change, a dependency failure, a resource
exhaustion).

### 5. Systemic Contributing Factors

Blameless factors that allowed the incident to occur or worsen. Always
present. AI-generated.

Addresses process gaps, missing safeguards, tooling limitations, and systemic
issues — **not individual people**. Examples: lack of staged rollout, missing
automated tests, insufficient monitoring, inadequate runbook coverage.

### 6. Action Items

Corrective actions with suggested owners and priorities. Always present.
AI-generated.

Each action item follows the `description | owner, priority` format parsed
from the LLM response. Priority levels:

| Priority | Meaning          |
|----------|------------------|
| P0       | Highest urgency  |
| P1       | Medium urgency   |
| P2       | Lowest urgency   |

The `ActionItem` model includes an `ai_generated: bool` field so each item is
labeled in the output. If the LLM produces no action items, a placeholder
item ("No specific action items identified.", owner: TBD, priority: P2) is
inserted so the section is never empty.

### 7. Stakeholder Communication Log

A log of all stakeholder updates sent during the incident. SEV1 only. Marked
as **session data** (not AI-generated) because it is reconstructed from sent
`StakeholderUpdate` records.

Lists each update with its number, timestamp, impact summary, and approval
status.

### 8. Regulatory/Compliance Impact

Compliance and regulatory notes. SEV1 only. AI-generated.

Addresses whether the incident triggered any regulatory obligations (e.g.
PCI-DSS notification requirements, GDPR breach reporting, SOX controls
impact). For incidents with no regulatory impact, the section notes that
explicitly.

## Postmortem Model

The `Postmortem` Pydantic model captures the full COE structure:

| Field                                | Type                          | Default | Description                              |
|--------------------------------------|-------------------------------|---------|------------------------------------------|
| `incident_id`                        | `str`                         | —       | Incident identifier                      |
| `incident_date`                      | `datetime`                    | —       | Alert timestamp                          |
| `severity`                           | `str`                         | —       | Incident severity (SEV1/SEV2/SEV3)       |
| `service`                            | `str`                         | —       | Affected service                         |
| `summary`                            | `PostmortemSection`           | —       | Section 1 — always present               |
| `timeline`                           | `PostmortemSection`           | —       | Section 3 — always present (session data)|
| `root_cause_analysis`                | `PostmortemSection`           | —       | Section 4 — always present               |
| `systemic_contributing_factors`      | `PostmortemSection`           | —       | Section 5 — always present               |
| `action_items`                       | `list[ActionItem]`           | —       | Section 6 — always present               |
| `customer_impact`                    | `PostmortemSection \| null`  | `null`  | Section 2 — SEV1/SEV2 only               |
| `stakeholder_communication_log`      | `PostmortemSection \| null`  | `null`  | Section 7 — SEV1 only (session data)     |
| `regulatory_compliance_impact`       | `PostmortemSection \| null`  | `null`  | Section 8 — SEV1 only                    |
| `resolved_at`                        | `datetime \| null`           | `null`  | Proxy: last timeline event timestamp     |
| `mttr_minutes`                       | `int \| null`                | `null`  | Minutes from alert to resolution         |
| `approved`                           | `bool`                        | `False` | Human review gate                        |

### `PostmortemSection` model

| Field          | Type   | Default | Description                                    |
|----------------|--------|---------|------------------------------------------------|
| `title`        | `str`  | —       | Section title                                  |
| `content`      | `str`  | —       | Section body text                              |
| `ai_generated` | `bool` | `True`  | `False` when authored/reconstructed by humans  |

### `ActionItem` model

| Field             | Type                   | Default | Description                          |
|-------------------|------------------------|---------|--------------------------------------|
| `description`     | `str`                  | —       | What needs to be done                |
| `suggested_owner` | `str`                  | —       | Recommended team or person           |
| `priority`        | `"P0" \| "P1" \| "P2"`| `"P1"`  | Urgency level                        |
| `ai_generated`    | `bool`                 | `True`  | `False` when human-authored          |

---

## MTTR Calculation

MTTR (Mean Time To Resolution) is computed as:

```
mttr_minutes = (resolved_at - alert.timestamp) / 60
```

`resolved_at` is a proxy value: the timestamp of the last timeline event. This
is the best approximation available from structured data. The actual
resolution timestamp may differ.

---

## Human-in-the-Loop Review

The `interrupt_for_postmortem_review` node provides a human review gate:

- **`simulate` mode** — auto-approves the postmortem (`approved = True`)
- **`run` mode** — pauses via LangGraph interrupt for human review before
  publishing

This gate exists so the postmortem can be checked for blameless compliance and
factual accuracy before it is published, especially for AI-generated sections.

## Example COE Postmortem (SEV1)

```markdown
# Postmortem: INC-2026-001
- **Service:** payment-service
- **Severity:** SEV1
- **Date:** 2026-07-13T14:05:00

## Summary *[AI-GENERATED — review carefully]*
At 14:05 UTC on 2026-07-13, the payment-service began experiencing
database connection pool exhaustion, causing payment success rates to
drop below 50%. The incident lasted 45 minutes and affected all global
checkout flows. The root cause was a configuration change in PR #4521
that reduced the connection pool size from 50 to 10.

## Customer Impact *[AI-GENERATED — review carefully]*
Approximately 12,000 checkout attempts failed during the 45-minute
incident window, affecting all global regions. An estimated $45,000 in
transactions was impacted. No data corruption occurred; all failures
were connection timeouts that retried safely.

## Timeline *[From session data]*
- [2026-07-13T13:45:00] [github] PR #4521 merged: Increase DB connection
  pool size
- [2026-07-13T14:05:00] [alert] Payment success rate dropped below 50%
- [2026-07-13T14:05:00] [log] Connection pool exhausted
- [2026-07-13T14:06:00] [chat] First responder noticed error spike
- [2026-07-13T14:10:00] [manual] IC declared
- [2026-07-13T14:20:00] [manual] Rollback started [DEPLOY]
- [2026-07-13T14:50:00] [manual] Error rate back to baseline

## Root Cause Analysis *[AI-GENERATED — review carefully]*
PR #4521 changed the DB connection pool size from 50 to 10 in
config/db-pool.yaml. The change was intended to reduce memory overhead
but was deployed without staged rollout. Under peak traffic, 10
connections were insufficient, causing pool acquire timeouts that
propagated as payment failures.

## Systemic Contributing Factors *[AI-GENERATED — review carefully]*
- No staged rollout requirement for configuration changes to critical
  services
- Connection pool size changes were not covered by automated integration
  tests
- Monitoring did not alert on pool utilization before exhaustion,
  reducing response lead time
- The runbook for DB pool issues existed but was not indexed in the RAG
  system at deploy time

## Action Items
- **Add integration test for pool size changes** — owner: Platform Team, priority: P0 *[AI-generated]*
- **Require staged rollout for config changes to critical services** — owner: SRE Team, priority: P1 *[AI-generated]*
- **Add pool utilization alerting at 80% threshold** — owner: Observability Team, priority: P1 *[AI-generated]*
- **Index all runbooks in RAG system** — owner: Platform Team, priority: P2 *[AI-generated]*

## Stakeholder Communication Log *[From session data]*
- Update 1 (14:06) — Impact: payment failures; Confidence: 65%
- Update 2 (14:12) — Impact: root cause identified; Confidence: 85%
- Update 3 (14:25) — Impact: rollback deployed, recovering; Confidence: 90%
- Update 4 (14:50) — Impact: resolved; Confidence: 95%

## Regulatory/Compliance Impact *[AI-GENERATED — review carefully]*
PCI-DSS compliance was not directly affected. No cardholder data was
exposed. However, 12,000 failed transactions may require customer
notification per incident response policy. Finance should review for
any SLA breach penalties with enterprise customers.

- **MTTR:** 45 minutes
```
